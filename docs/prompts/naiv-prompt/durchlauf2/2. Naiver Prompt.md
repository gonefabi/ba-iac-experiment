### Analysierte Terraform-Zustandsprobleme und identifizierte Lösungsoptionen

Das ist ein anderes Problem als zuvor. Diesmal ist es kein SKU-Fehler, sondern ein **State-Konflikt**: Die Ressourcengruppe `firma-rg` (und die Netzwerk-Ressourcen darin) existieren noch von den vorherigen Läufen in Azure, aber der Terraform-State kennt sie nicht — vermutlich weil der State in deiner GitHub-Actions-Pipeline nicht zwischen Runs persistiert wird.

Du hast zwei Möglichkeiten:

**Option A — Sauberer Neustart (empfohlen):** Lösche die bestehende Ressourcengruppe in Azure, dann läuft der nächste `terraform apply` sauber durch:

```bash
az group delete --name firma-rg --yes --no-wait
```

Warte ein paar Minuten, bis Azure die Löschung abgeschlossen hat, dann starte die Pipeline erneut.

**Option B — Remote-Backend einrichten**, damit der State zwischen Pipeline-Runs erhalten bleibt. Das verhindert dieses Problem dauerhaft. Dazu bräuchtest du einen Azure Storage Account als Backend:

```hcl
terraform {
  backend "azurerm" {
    resource_group_name  = "tfstate-rg"
    storage_account_name = "tfstatefirma"
    container_name       = "tfstate"
    key                  = "infra.terraform.tfstate"
  }
}
```

Für den Moment empfehle ich Option A — die Ressourcengruppe löschen und neu deployen. Damit das Problem langfristig nicht wiederkommt, lohnt sich Option B als nächster Schritt. Soll ich dir die Terraform-Konfiguration inklusive Remote-Backend-Setup ergänzen?
