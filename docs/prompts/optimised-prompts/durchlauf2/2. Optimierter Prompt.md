## Benutzer

Erstelle eine vollständige, produktionsreife Terraform-Konfiguration für Microsoft Azure mit dem AzureRM-Provider (Version >= 4.0).

**Wichtiger Hinweis zur Ressourcengruppe**

Die Ressourcengruppe "rg-kmu-experiment" existiert bereits in Azure und darf NICHT von Terraform erstellt oder gelöscht werden, da dort Azure Policies als Governance-Kontrollinstanz hinterlegt sind. Referenziere sie ausschließlich per data source: data "azurerm_resource_group" "main" { name = "rg-kmu-experiment" } Alle anderen Ressourcen sollen erstellt werden. Ersetze alle Referenzen auf azurerm_resource_group.main durch data.azurerm_resource_group.main

**Pflicht-Tags (auf JEDER einzelnen Ressource)**

Jede Ressource muss exakt diese drei Tags tragen: Environment = "Development" Owner = "Fabian Klein" Project = "BA-IaC-Experiment"

**Namenskonventionen (exakt einhalten)**

* Virtual Network: vnet-kmu-main
* Subnetz 1: snet-user
* Subnetz 2: snet-app
* NSG 1: nsg-user
* NSG 2: nsg-app
* Windows VM: vm-kmu-jumpbox
* Linux VM: vm-kmu-app
* Public IP: pip-vm-kmu-jumpbox
* NIC Windows: nic-vm-kmu-jumpbox
* NIC Linux: nic-vm-kmu-app

**Netzwerk**

* VNet-Adressraum: 10.0.0.0/16
* snet-user: 10.0.1.0/24
* snet-app: 10.0.2.0/24
* NSGs müssen über azurerm_subnet_network_security_group_association auf Subnetz-Ebene zugeordnet werden (NICHT auf NIC-Ebene)

**Sicherheitsregeln NSG nsg-user**

* Inbound ALLOW: RDP (TCP Port 3389) von var.admin_source_ip/32
* Inbound DENY: Alle anderen (Priority 4096)
* Outbound: Allow All

**Sicherheitsregeln NSG nsg-app**

* Inbound ALLOW: SSH (TCP Port 22) nur von 10.0.1.0/24, Priority 100
* Inbound ALLOW: HTTP (TCP Port 80) nur von 10.0.1.0/24, Priority 110
* Inbound DENY: Alle anderen (Priority 4096)
* Outbound: Allow All

**Variablen (exakte Namen Pflicht)**

* windows_admin_password (string, sensitive)

**Codestruktur**

Erstelle drei separate Dateien:

* main.tf → Provider, data source, alle Ressourcen
* variables.tf → Alle Variablen mit Defaults
* outputs.tf → Public IP Windows-VM, Private IPs beider VMs

**Technische Pflichtanforderungen**

* azurerm_linux_virtual_machine und azurerm_windows_virtual_machine (nicht das veraltete azurerm_virtual_machine)
* Keine deprecated Ressourcentypen
* Provider-Block mit features {} ist Pflicht
* Region: germanywestcentral
* Kein Remote-Backend (lokaler State)

## Claude

Orchestrierte produktionsreife Azure-Infrastruktur mit strikten Governance-Anforderungen

Ich erstelle die drei Terraform-Dateien mit allen Spezifikationen.

Orchestrierte umfassende Azure-Terraform-Konfiguration mit Governance-Anforderungen

Die drei Terraform-Dateien sind fertig. Hier eine Zusammenfassung der Architektur:

**main.tf** enthält den gesamten Infrastrukturcode: Provider-Block mit features {}, die bestehende Ressourcengruppe als data-Source (wird weder erstellt noch gelöscht), VNet mit zwei Subnetzen, zwei NSGs mit Subnetz-Assoziation, eine Windows-Jumpbox mit Public IP im snet-user und eine Linux-App-VM ohne Public IP im snet-app. Alle Ressourcen tragen die drei Pflicht-Tags via local.mandatory_tags.

**variables.tf** definiert die drei geforderten Variablen: windows_admin_password und linux_admin_password (beide sensitive = true) sowie admin_source_ip (default "*").

**outputs.tf** gibt die öffentliche IP der Jumpbox sowie die privaten IPs beider VMs aus.

Zum Deployen legst du die drei Dateien in ein Verzeichnis und führst aus:

```bash
terraform init
terraform plan -var="windows_admin_password=DeinPasswort123!" \
               -var="linux_admin_password=DeinPasswort456!" \
               -var="admin_source_ip=203.0.113.42"
terraform apply
```

Ein Hinweis zur admin_source_ip: Die Variable wird im NSG mit /32 ergänzt. Falls du den Default "*" nutzt, wird */32 gesetzt — für Produktivbetrieb solltest du immer eine konkrete IP angeben.
